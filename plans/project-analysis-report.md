# Сводный отчет о проекте PoseFlow Editor

## Общая информация

**Название проекта:** PoseFlow Editor  
**Версия:** v0.2.0  
**Тип:** Десктопное приложение для редактирования 3D поз  
**Целевая аудитория:** Художники, использующие ControlNet/Stable Diffusion  
**Основной язык:** Русский (UI, документация, коммиты)  
**Статус:** Активная разработка, реализованы шаги 1-7, Phase 4 в процессе реализации  

## Архитектурный обзор

### Трехслойная архитектура
```
React (Vite) ←IPC→ Electron ←HTTP→ Python FastAPI
```

**Слой 1: React фронтенд (TypeScript)**
- Three.js через @react-three/fiber/drei
- Модульная структура: components/, services/, lib/, hooks/
- Состояние: PoseService (синглтон) + AppSettingsContext + Zustand stores (экспериментально)
- DI контейнер готов к активации

**Слой 2: Electron оболочка**
- IPC мост между React и Python
- Управление жизненным циклом Python процесса
- Предоставление нативных API (диалоги, информация о системе)

**Слой 3: Python FastAPI бэкенд**
- Экспорт поз в форматы OpenPose (PNG/JSON)
- Микросервисная архитектура: ExportService, PoseService
- Отдельный процесс, управляемый Electron

### Ключевые компоненты

#### Сервисы
1. **PoseService** - центральный менеджер поз
   - Управление данными скелета BODY_25
   - Undo/Redo стек (максимум 50 действий)
   - FK/IK режимы манипуляции
   - Поддержка множественных скелетов (заготовка для Step 9)

2. **ExportService** - экспорт в OpenPose форматы
   - Проекция 3D в 2D с учетом камеры
   - Обрезка по рамке (алгоритм Liang-Barsky)
   - Поддержка разрешений 1K/2K/4K

3. **SkeletonGraph** - графовая модель скелета
   - Иерархия BODY_25 с корнем в MID_HIP
   - Вычисление длин костей
   - FK пропагация с поддержкой unlinked суставов

#### Алгоритмы
1. **FABRIK IK solver** - инверсная кинематика для 4 цепей
   - Руки (плечо-запястье)
   - Ноги (бедро-лодыжка)
   - Сходимость за 10-20 итераций

2. **Camera-plane drag** - манипуляция суставами в плоскости камеры
   - Raycasting для определения плоскости
   - Плавное перемещение с учетом FK/IK режима

## Анализ архитектурных решений

### Сильные стороны
1. **Четкое разделение ответственности:** Логика отделена от UI, алгоритмы в lib/
2. **Модульность:** Сервисы независимы, легко тестируются
3. **Расширяемость:** Подготовка для множественных скелетов уже заложена
4. **Кроссплатформенность:** Electron + Python обеспечивают работу на Windows/macOS/Linux
5. **Офлайн-работа:** Полная автономность после установки
6. **Feature Flags:** Безопасное внедрение экспериментальных функций
7. **DI Container:** Готовность к переходу на инжектирование зависимостей

### Архитектурные решения (ADR)
1. **ADR-0001:** Выбор FABRIK для IK вместо CCD или аналитических методов
   - Причина: Простота реализации, хорошая сходимость для цепей BODY_25
   - Последствия: Нужна проверка на вырожденные случаи

2. **ADR-0002:** Согласование проекции камеры для PNG экспорта
   - Причина: Обеспечение соответствия между 3D видом и 2D экспортом
   - Последствия: Требует синхронизации матриц проекции

## Производительность и ограничения

### Известные ограничения
1. **Нет ограничений длины костей** при FK drag (возможен stretch) - решается FixedLengthSolver
2. **Нет биомеханических ограничений** на вращения суставов - в разработке
3. **Одноэлементный PoseService** может стать узким местом при масштабировании - решается DI контейнером
4. **Python процесс** должен быть запущен для экспорта
5. **Экспериментальные компоненты** требуют интеграции и оптимизации производительности

### Производительность
- **Рендеринг:** 60 FPS на современных системах
- **IK вычисления:** < 1 мс на цепь (4 цепи = ~4 мс)
- **Память:** ~50-100 МБ для основного процесса
- **Загрузка:** Холодный старт ~3-5 секунд (запуск Python + Electron)
