# Интегрированный план улучшений: Архитектура + DesignDoll рефакторинг

## Обзор

Этот документ объединяет два ключевых направления улучшений:
1. **Архитектурные улучшения** (Dependency Injection, State Management, оптимизация)
2. **Рефакторинг управления** в стиле DesignDoll (фиксированные длины, контроллеры, модель A+B)

Цель: обеспечить максимально безболезненный и надежный переход на новую систему управления с параллельным улучшением архитектуры.

## Принципы интеграции

### 1. Поэтапность
Каждое изменение должно быть инкрементальным и обратно совместимым. Новые функции вводятся через feature flags.

### 2. Параллельные потоки
- **Поток A:** Архитектурные улучшения (фундамент)
- **Поток B:** DesignDoll рефакторинг (функциональность)
- Оба потока синхронизируются на ключевых этапах

### 3. Тестирование на каждом шаге
Каждое изменение сопровождается:
- Unit тестами для новой функциональности
- Регрессионными тестами для существующей
- Интеграционными тестами для взаимодействия компонентов

## Синхронизированный план

### Фаза 0: Подготовка (2 недели)

**Общие задачи:**
1. Создание ветки `refactor/integrated`

4. Создание baseline тестов для регрессии

**Архитектурные улучшения:**
- Настройка DI контейнера (без внедрения в production код)

**DesignDoll подготовка:**
- Анализ текущего поведения drag системы
- Создание прототипа фиксированных длин в изолированном модуле

**Поток A: Архитектурные улучшения**

   - Создание `ServiceContainer` с поддержкой как старых, так и новых сервисов

2. **State Management базовый слой**
   - Внедрение Zustand для UI состояния
   - Сохранение обратной совместимости с текущим PoseService

**Поток B: DesignDoll - Фаза A1/A2**
1. **Жесткая фиксация длины кости**

**Интеграционная точка:**

### Фаза 2: Контроллеры и состояние (4 недели)

**Поток A: Расширение архитектуры**
1. **Полное внедрение DI**

1. **Жесткий череп**

   - Прототип `SpineChain` с виртуальными сегментами
   - Двунаправленный маппинг между виртуальным состоянием и BODY_25

3. **Семь контроллеров**
   - Реестр `MainControllers`
   - Визуальное выделение контроллеров в UI
   - Адаптер для преобразования drag событий

**Интеграционная точка:**
- State store управляет как старой позой (PoseData), так и новой (ExtendedPoseData)
- DI контейнер предоставляет соответствующие сервисы в зависимости от активных feature flags

### Фаза 3: Модель A+B и оптимизация (3 недели)

**Поток A: Оптимизация производительности**

   - Виртуализация для множественных скелетов

2. **Оптимизация алгоритмов**
   - Кэширование для FixedLengthSolver
   - Профилирование и оптимизация FABRIK с ограничениями

**Поток B: DesignDoll - Фаза C**
1. **Mini-view (Step 8)**
   - Реализация как отдельный компонент
   - Синхронизация камеры через общий store

2. **Направление B (Step 11)**
   - Кольцевые гизмо как отдельный режим взаимодействия
   - Интеграция с системой ограничений

3. **Конечный автомат взаимодействия**
   - Унифицированная система состояний
   - Поддержка как старого, так и нового режимов управления

### Фаза 4: Консолидация и переход (2 недели)

**Общие задачи:**
1. **Постепенное включение feature flags**
   - Сначала в dev окружении
   - Затем для beta-тестеров
   - Наконец, для всех пользователей

2. **Миграция данных**
   - Конвертер старых пресетов в новую модель
   - Обратная совместимость для сохраненных поз

3. **Удаление старого кода**
   - После стабилизации новой системы
   - Поэтапное удаление feature flags и старого кода

### 2. Адаптивная система сервисов

```typescript
class AdaptivePoseService implements IPoseService {
  constructor(
    private legacyService: LegacyPoseService,
    private ddService: DesignDollPoseService,
    private featureFlags: FeatureFlags
  ) {}
  
  getPoseData(): PoseData {
    if (this.featureFlags.useDesignDollMode) {
      const extended = this.ddService.getExtendedPoseData();
      return PoseConverter.toLegacy(extended);
    }
    return this.legacyService.getPoseData();
  }
  
  // Аналогично для других методов
}
```

### 3. Feature Flags система

```typescript
const FEATURE_FLAGS = {
  USE_FIXED_LENGTHS: {
    enabled: false,
    rolloutPercentage: 0, // 0-100
    canaryUsers: ['test1', 'test2']
  },
  USE_RIGID_SKULL: {
    enabled: false,
    dependsOn: ['USE_FIXED_LENGTHS']
  },
  USE_DESIGNDOLL_MODE: {
    enabled: false,
    dependsOn: ['USE_FIXED_LENGTHS', 'USE_RIGID_SKULL', 'USE_SPINE_CHAIN']
  }
};
```

## Риски и митигации

### Риск 1: Сложность поддержки двух систем одновременно
**Митигация:**
- Четкое разделение кода через интерфейсы
- Автоматические тесты на конвертацию данных
- Регулярная синхронизация между ветками

### Риск 2: Производительность при dual-mode
**Митигация:**
- Ленивая инициализация новых систем
- Отложенная загрузка тяжелых модулей
- Профилирование на каждом этапе

### Риск 3: Регрессии в экспорте
**Митигация:**
- Golden tests для экспорта ключевых поз
- Визуальное сравнение до/после
- Canary deployment для экспортного функционала

## Заключение

Интегрированный план обеспечивает:
1. **Безболезненный переход:** Поэтапное внедрение с сохранением обратной совместимости
2. **Надежность:** Comprehensive testing на каждом этапе
3. **Параллельное развитие:** Архитектурные улучшения и функциональный рефакторинг идут одновременно
4. **Гибкость:** Feature flags позволяют контролировать rollout и откатывать проблемные изменения

Ключевой успех зависит от:
- Строгого соблюдения интерфейсов
- Регулярной синхронизации между потоками
- Непрерывного тестирования и мониторинга
