# План ручного тестирования PoseFlow

> Legacy note, 2026-04-28: этот план содержит исторические тест-кейсы для старых Steps/Phase 4 и может ссылаться на удалённые или отложенные feature flags. Для текущей R3F-сцены, rotation-tree контроллеров, drag, mini-view и export frame использовать актуальный чек-лист: `ai/docs/r3f-smoke-manual-checklist.md`.

## Обзор
Этот документ содержит структурированный план ручного тестирования для проекта PoseFlow. План предназначен для совместного тестирования, где тестировщик выполняет тесты, а координатор отслеживает результаты.

## Цели тестирования
1. Проверить корректность работы базового функционала (Steps 1-7)
2. Проверить работоспособность новых функций Phase 4 (Steps 8-11)
3. Проверить архитектурные компоненты (feature flags, DI контейнер, state management)
4. Проверить экспериментальные модули DesignDoll
5. Проверить интеграционное взаимодействие компонентов

## Структура тестирования
Тесты организованы в логической последовательности от простого к сложному, с учетом зависимостей между компонентами.

## 1. Базовый функционал (Steps 1-7)

### 1.1. Step 1: Camera-plane drag
**Цель:** Проверить возможность перетаскивания суставов в плоскости камеры
**Приоритет:** Высокий

**Тест-кейсы:**
1. **TC-BASE-001:** Перетаскивание сустава в плоскости камеры
   - Шаги: 
     1. Запустить приложение
     2. Выбрать сустав (например, правую кисть)
     3. Перетащить сустав в плоскости камеры
   - Ожидаемый результат: Сустав перемещается плавно, остальные суставы сохраняют позиции
   - Критерии успеха: Перемещение происходит без задержек, позиция обновляется в реальном времени

2. **TC-BASE-002:** Перетаскивание с изменением угла камеры
   - Шаги:
     1. Изменить угол камеры (использовать компас камеры)
     2. Перетащить сустав в новой плоскости
   - Ожидаемый результат: Перетаскивание корректно работает в любой ориентации камеры
   - Критерии успеха: Сустав перемещается в плоскости текущей камеры

### 1.2. Step 2: Undo/Redo + PoseService
**Цель:** Проверить систему отмены/повтора действий
**Приоритет:** Высокий

**Тест-кейсы:**
3. **TC-BASE-003:** Отмена перемещения сустава
   - Шаги:
     1. Переместить сустав
     2. Нажать Ctrl+Z или кнопку Undo
   - Ожидаемый результат: Сустав возвращается в исходное положение
   - Критерии успеха: История действий сохраняется корректно

4. **TC-BASE-004:** Повтор отмененного действия
   - Шаги:
     1. Переместить сустав
     2. Отменить действие (Ctrl+Z)
     3. Повторить действие (Ctrl+Y или кнопка Redo)
   - Ожидаемый результат: Сустав возвращается в перемещенное положение
   - Критерии успеха: Redo восстанавливает последнее отмененное действие

### 1.3. Step 3: SkeletonGraph + FK
**Цель:** Проверить прямую кинематику (FK) и граф скелета
**Приоритет:** Высокий

**Тест-кейсы:**
5. **TC-BASE-005:** FK перемещение цепочки суставов
   - Шаги:
     1. Перетащить сустав в иерархии (например, плечо)
     2. Наблюдать за дочерними суставами (локоть, кисть)
   - Ожидаемый результат: Дочерние суставы следуют за родительским с сохранением относительных позиций
   - Критерии успеха: Иерархия сохраняется корректно

### 1.4. Step 4: FABRIK IK solver
**Цель:** Проверить инверсную кинематику (IK)
**Приоритет:** Высокий

**Тест-кейсы:**
6. **TC-BASE-006:** IK перемещение конечного эффектора
   - Шаги:
     1. Включить IK режим (если есть переключатель)
     2. Перетащить конечный эффектор (кисть)
     3. Наблюдать за поведением всей цепи (плечо-локоть-кисть)
   - Ожидаемый результат: Вся цепь автоматически сгибается для достижения целевой позиции
   - Критерии успеха: Алгоритм FABRIK работает корректно, нет разрывов в цепи

### 1.5. Step 5: FK/IK mode toggle UI
**Цель:** Проверить переключение между режимами FK и IK
**Приоритет:** Средний

**Тест-кейсы:**
7. **TC-BASE-007:** Переключение режимов FK/IK
   - Шаги:
     1. Найти переключатель режимов в UI
     2. Переключиться из FK в IK
     3. Перетащить сустав
     4. Вернуться в FK режим
   - Ожидаемый результат: Поведение перетаскивания меняется соответственно режиму
   - Критерии успеха: Визуальная индикация режима работает корректно

### 1.6. Step 6: Pose mirroring
**Цель:** Проверить зеркальное отражение позы
**Приоритет:** Средний

**Тест-кейсы:**
8. **TC-BASE-008:** Зеркальное отражение позы (кнопка)
   - Шаги:
     1. Создать асимметричную позу
     2. Нажать кнопку Mirror или клавишу M
   - Ожидаемый результат: Поза зеркально отражается (левая и правая стороны меняются местами)
   - Критерии успеха: Симметрия сохраняется корректно

### 1.7. Step 7: Link/Unlink joints
**Цель:** Проверить связывание и развязывание суставов
**Приоритет:** Средний

**Тест-кейсы:**
9. **TC-BASE-009:** Связывание суставов через правый клик
   - Шаги:
     1. Правый клик на суставе
     2. Выбрать опцию Link/Unlink
     3. Проверить поведение при перетаскивании
   - Ожидаемый результат: Связанные суставы двигаются вместе
   - Критерии успеха: Состояние связи сохраняется между действиями

## 2. Новый функционал Phase 4 (Steps 8-11)

    - Критерии успеха: Группы суставов сохраняют относительные позиции

### 2.4. Step 11: Ring gizmos (ротационные контроллеры)

      2. Двойной клик на суставе для показа гизмо
      3. Перетащить одно из колец для вращения

    - Критерии успеха: Вращение происходит плавно, оси соответствуют цветам (X-красный, Y-зеленый, Z-синий)

## 6. Система отчетности

### 6.1. Формат записи результатов
Для каждого теста необходимо заполнить следующую информацию:

```
Тест: [ID теста] - [Название теста]
Дата: [Дата выполнения]
Тестировщик: [Имя тестировщика]
Версия приложения: [Версия PoseFlow]

Шаги выполнения:
1. [Шаг 1]
2. [Шаг 2]
3. [Шаг 3]

Результат: [Пройден/Не пройден/Блокирован]
Ожидаемый результат: [Что ожидалось]
Фактический результат: [Что произошло на самом деле]

Примечания: [Любые дополнительные наблюдения]
Скриншоты: [Ссылки на скриншоты, если есть]
```

### 6.2. Шаблон для документирования ошибок
```
Баг-репорт: [Краткое описание]
ID теста: [Связанный тест]
Серьезность: [Критический/Высокий/Средний/Низкий]
Приоритет: [Высокий/Средний/Низкий]

Шаги воспроизведения:
1. [Шаг 1]
2. [Шаг 2]
3. [Шаг 3]

Ожидаемое поведение: [Что должно было произойти]
Фактическое поведение: [Что произошло на самом деле]

Частота: [Всегда/Иногда/Редко]
Окружение: [ОС, браузер, версия приложения]
Скриншоты/Логи: [Приложить если есть]
```

### 6.3. Процесс эскалации проблем
1. **Уровень 1:** Тестировщик документирует проблему в шаблоне баг-репорта
2. **Уровень 2:** Координатор проверяет и подтверждает проблему
3. **Уровень 3:** Проблема передается разработчику для исправления
4. **Уровень 4:** После исправления тестировщик выполняет регрессионное тестирование

## 7. Инфраструктура для совместного тестирования

### 7.1. Чек-листы для пользователя
**Чек-лист подготовки к тестированию:**
- [ ] Установить последнюю версию PoseFlow
- [ ] Убедиться, что все зависимости установлены
- [ ] Проверить доступность backend сервиса
- [ ] Очистить кэш браузера/приложения
- [ ] Подготовить скриншот-инструмент

**Чек-лист выполнения тестов:**
- [ ] Выбрать категорию тестов для выполнения
- [ ] Следовать шагам каждого тест-кейса
- [ ] Записывать результаты по шаблону
- [ ] Делать скриншоты при обнаружении проблем
- [ ] Сообщать о завершении категории координатору

### 7.2. Инструкции по запуску тестов
1. **Запуск приложения:**
   ```bash
   cd poseflow
   npm install
   npm run dev
   ```

2. **Запуск backend:**
   ```bash
   cd poseflow/backend
   python -m venv venv
   venv\Scripts\activate  # Windows
   pip install -r requirements.txt
   python main.py
   ```

3. **Доступ к feature flags:**
   - В dev режиме панель feature flags доступна в правом верхнем углу
   - Для production тестирования использовать URL параметры или конфигурационные файлы

### 7.3. Скрипты для подготовки тестовой среды
Создать файл `setup-test-environment.bat`:
```batch
@echo off
echo Подготовка тестовой среды PoseFlow...

cd /d "d:\ai\QwenCoder\poseflow"
echo Очистка node_modules...
rmdir /s /q node_modules 2>nul

echo Установка зависимостей...
npm install

echo Запуск backend в фоновом режиме...
start cmd /k "cd backend && python main.py"

echo Запуск frontend...
start cmd /k "npm run dev"

echo Тестовая среда готова!
pause
```

## Процесс совместного тестирования

### Роли:
1. **Координатор тестирования:** Создает задания, отслеживает прогресс, анализирует результаты
2. **Тестировщик:** Выполняет тесты, документирует результаты, сообщает о проблемах

### Workflow:
1. Координатор назначает категорию тестов для выполнения
2. Тестировщик выполняет тесты из категории
3. Тестировщик заполняет отчеты по каждому тесту
4. Координатор проверяет отчеты и создает баг-репорты при необходимости
5. Процесс повторяется для следующей категории

### Инструменты для совместной работы:
- **Для отчетов:** Google Sheets/Excel с шаблонами
- **Для баг-трекинга:** GitHub Issues, Jira, или простой текстовый файл
- **Для коммуникации:** Slack, Discord, или мессенджер
- **Для скриншотов:** Lightshot, Snipping Tool + облачное хранилище
