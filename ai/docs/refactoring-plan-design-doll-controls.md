## 1. Цели и критерии готовности

**Цели**

- Пользователь выставляет позу в **логике DesignDoll**: торс → плечи → кисти/стопы (точка A + направление B) → баланс → мелкая правка; глубина опирается на **мини-вид**.
- **Фиксированные длины костей** при обычном редактировании; исключение — **гибкость цепи шея + позвоночник** (несколько степеней свободы между существующими ключевыми точками BODY_25 или виртуальными сегментами).
- **Череп**: точки головы/лица — **жёсткая группа** относительно шеи; нет отдельного «гнуть лоб».
- **Семь главных контроллеров** (приоритетный ввод): основание черепа, верхняя треть позвоночника, крестец (связка с тазом/MID_HIP), оба запястья, обе лодыжки.
- **Биомеханика**: часть узлов без пользовательского вращения; часть — с **клиньями** углов (таблица лимитов).

**Критерии «готово» (MVP рефакторинга)**

- Можно собрать типовую позу из Practice-туториала (сидячая, наклон, руки к бёдрам) без растягивания костей.
- Запястья и лодыжки управляются как **контроллеры** с отдельным шагом «направление B» (кольца или эквивалент).
- Мини-вид помогает глубине; основной вид — camera-plane drag без потери согласованности с экспортом.

## 2. Зависимости от текущего кода

| Компонент | Сейчас | Роль в рефакторинге |
|-----------|--------|---------------------|
| `SkeletonGraph` | FK, длины костей, unlink | База; добавить жёсткое соблюдение длины при drag, виртуальный позвоночник |
| `FABRIKSolver` / IK chains | 4 цепи руки/нога | Расширить под лимиты, при необходимости доп. цепи |
| `useTransformDrag` | Плоскость камеры | Остаётся для «поверхностного» drag; контроллеры и кольца — дополнительные режимы |
| `Joint.tsx` | Сфера + drag | Разделить: обычный сустав vs главный контроллер vs гизмо |
| `PoseService` | Снимки, undo | Расширить тип состояния (виртуальные DOF позвоночника) |
| Экспорт | `projectTo2D` | Не ломать; поза всё так же BODY_25 |

## 3. Фазы работ (рекомендуемый порядок)

### Фаза A — Модель данных и инварианты

**A1. Жёсткая фиксация длины кости**

- При любом изменении позиции сустава решать: **обновление дочерних/родительских** так, чтобы длины рёбер графа оставались в допуске (или явный IK-проход).
- Устранить известный gap: «растягивание при FK drag» ([`STATUS.md`](../../STATUS.md)).
- **Тесты**: чистые функции нормализации цепочки фиксированной длины, граничные случаи (180°, вырожденный треугольник).

**A2. Таблица допустимых DOF по суставам**

- Новый модуль, например `src/lib/body25/jointLimits.ts`: для каждого `Body25Index` — какие оси/углы разрешены, запрет вращения, `min/max` или клинья в локальных осях.
- Связка с IK: цель недостижима → ближайшая допустимая конфигурация (clamp).

**A3. Жёсткий череп**

- Задать группу индексов «голова» (NOSE, EYES, EARS, …) и правило: их позиции = **жёсткое преобразование** от опорной точки «основание черепа» (новая виртуальная или производная от NECK/NOSE).
- **Тесты**: преобразование группы при повороте/сдвиге базы черепа.

**A4. Цепь позвоночника и шеи (гибкость)**

- Ввести **внутренние** сегменты (например 2–4 виртуальных шарнира между NECK ↔ MID_HIP и NECK ↔ голова-группа) с хранением в состоянии позы (расширение `PoseData` или отдельный `SpineState`).
- Экспорт в OpenPose: по-прежнему **25 точек** — финальные координаты получать **скиннингом/упрощённым FK** из виртуальной цепи в видимые BODY_25 точки.
- Это самая рискованная подфаза: отдельный прототип и согласование с визуалом костей.

---

### Фаза B — Семь контроллеров и приоритет ввода

**B1. Реестр контроллеров**

- Константа `MAIN_CONTROLLERS: Body25Index[]` + виртуальные id для «основание черепа» и «верхняя треть позвоночника», если они не совпадают с одной точкой.
- UI: визуальное отличие (размер, цвет, опционально только они кликабельны в «режиме контроллеров»).

**B2. Поведение drag по контроллеру**

- Перемещение крестца/таза: влияет на нижнюю часть тела и якорь IK ног.
- Запястья/лодыжки: позиция **A** (camera-plane или с привязкой к лимиту длины цепи).

**B3. Связь с существующими шагами PLAN**

- Step 10 (CoG) в DesignDoll — сдвиг «центра масс» груди/таза; у нас уже запланирован `CenterOfGravity.tsx` — **встроить после** базовых контроллеров и фиксированных длин.

### Фаза C — Модель «A + B» и мини-вид

**C1. Mini-view (Step 8 из PLAN)**

- Второй канвас 90° к основному; без взаимодействия; синхронизация камеры — как в `PLAN.md`.
- Блокирует UX-парадокс «глубина в 2D» из PoseTag.

**C2. Направление B**

- Step 11 (кольца) — реализовать как **режим вторичного ввода** после выбора контроллера кисти/стопы (и при необходимости плеча/бедра): кольца вокруг локальной системы конечности.
- Альтернатива MVP: 2D-манипулятор «компас» в UI вместо трёх торов — зафиксировать в дизайне.

**C3. Единый конечный автомат взаимодействия**

- Состояния: `navigate` | `drag_joint` | `drag_controller` | `adjust_direction_B` | `orbit_camera`.
- Чётко разделить ПКМ/ЛКМ/двойной клик (как в текущем плане гизмо).

## 5. Риски и смягчение

| Риск | Смягчение |
|------|-----------|
| Виртуальный позвоночник ломает экспорт/пресеты | Двунаправленный маппинг «внутреннее состояние → 25 точек» + тесты на круговой round-trip |
| Слишком жёсткие лимиты → IK не сходится | Мягкие лимиты (штраф) или увеличение итераций FABRIK + fallback FK |
| Рост сложности UI | Поэтапно: сначала 7 контроллеров + фиксированные длины без полного позвоночника |
| Регрессии производительности | Профилирование `Skeleton3D`; не пересчитывать весь граф на каждый `pointermove` без debounce где возможно |

## 6. Тестирование

- **Юнит-тесты** (обязательно): лимиты углов, фиксированная длина сегментов, жёсткая группа черепа, маппинг spine state → `PoseData`.
- **Ручная проверка**: сценарии из раздела Practice спецификации DesignDoll (наклон торса, сиденье, руки к бедру, баланс).
- Регрессия: экспорт PNG/JSON совпадает с видом в пределах допуска.

---

## 7. Что сознательно не входит в этот рефакторинг

- Morphing / Clay / Sketch / Memo / Import mesh (см. спецификацию).
- Пальцы (HandTag detail) — вне BODY_25.
- Fake perspective камеры — низкий приоритет.
