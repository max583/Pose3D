# План улучшений проектирования и разработки кода (PoseFlow)

Документ фиксирует договорённости по внедрению практик, дополняющих существующие правила в `CLAUDE.md`, `PLAN.md` и `STATUS.md`. Цель — предсказуемее проектирование, меньше архитектурного дрейфа и проще онбординг без раздувания бюрократии.

---

## 1. Цели

- Важные архитектурные решения **объяснимы и находимы** через год.
- Бэклог фич (шаги PLAN, рефакторинг DesignDoll) **не противоречит** приоритетам и зависимостям.
- Критерии готовности фичи **явные** (код, тесты, ручная проверка, документы).
- Чистая логика остаётся в `src/lib/` / `src/services/` с тестами; UI не «съедает» алгоритмы ради обхода правил тестирования.
- История изменений для пользователей и отладки — **отслеживаемая** (что вышло и когда).

---

## 2. Текущая опора (не отменяем)

- Описание стека и модулей: `CLAUDE.md`.
- Пошаговая реализация редактора: `PLAN.md`, архив завершённых шагов.
- Статус продукта: `STATUS.md`.
- Спецификация DesignDoll и план рефакторинга: `ai/docs/design-doll-controls-poseflow-spec.md`, `ai/docs/refactoring-plan-design-doll-controls.md`.
- Юнит-тесты для чистой логики, co-location `__tests__/`, прогон `npm test` в `poseflow/`.

---

## 3. Пакеты улучшений

### P0 — быстрый эффект, минимум формализма

**Статус: внедрено** — см. `CLAUDE.md` (разделы **Definition of Done**, **Git commit messages**), [`backlog-matrix.md`](./backlog-matrix.md), ссылка из [`PLAN.md`](../../PLAN.md).

| ID | Улучшение | Действие | Результат |
|----|-----------|----------|-----------|
| P0-1 | Definition of Done (DoD) | Зафиксировать короткий чеклист для «фича готова» (ниже §4) и применять при закрытии задач/PR | Меньше недоделанных релизов в `master` |
| P0-2 | Связка бэклогов | Одна таблица: шаги PLAN ↔ фазы DD-рефакторинга ↔ блокировки (файл `ai/docs/backlog-matrix.md` или раздел в начале `PLAN.md`) | Нет расхождения «делаем Step 8 vs фаза A» |
| P0-3 | Соглашение о коммитах | Описать в `CLAUDE.md` или в `ai/docs/` префиксы: `feat:`, `fix:`, `test:`, `docs:` + при необходимости ссылка на шаг (`PLAN step 8`, `DD фаза A1`) | Читаемый `git log` |

**Срок внедрения:** одна итерация (обновление документов + первая заполненная таблица бэклога).

---

### P1 — проектирование и сопровождение

| ID | Улучшение | Действие | Результат |
|----|-----------|----------|-----------|
| P1-1 | ADR (Architecture Decision Records) | Каталог `ai/decisions/` (или `docs/adr/`), шаблон: контекст → решение → последствия → статус (принято / устарело) | Объяснимость IK, экспорта, будущей модели позвоночника/лимитов |
| P1-2 | CHANGELOG | Файл `CHANGELOG.md` в корне репозитория или в `poseflow/`, формат [Keep a Changelog](https://keepachangelog.com/); обновлять при пользовательски заметных изменениях | Проще ответить «когда сломалось» и что нового |
| P1-3 | Граница логика/UI | В `CLAUDE.md` одним абзацем: если в компоненте появляется нетривиальный алгоритм — кандидат на перенос в `lib/` + тест | Меньше дрейфа в сторону «нетестируемого React» |

**Срок внедрения:** 2–3 итерации: сначала шаблон ADR + 1–2 записи по уже принятым решениям; затем CHANGELOG с текущей версией из `STATUS.md`.

---

### P2 — зрелость процесса (по необходимости)

| ID | Улучшение | Действие | Результат |
|----|-----------|----------|-----------|
| P2-1 | Теги релизов | Семантические теги `v0.x.y` при значимых вехах; в `CHANGELOG` — секция под тег | Связь кода и бинарей/дистрибутива |
| P2-2 | Шаблон PR | Короткий чеклист: соответствует DoD, затронуты тесты, обновлён CHANGELOG/STATUS при нужде | Единообразие ревью |

**Срок:** после стабилизации P0–P1 или перед публичным релизом.

---

## 4. Definition of Done (рекомендуемый минимум)

Перед слиянием фичи в основную ветку (адаптировать под масштаб задачи):

1. Код соответствует задаче; нет посторонних рефакторингов без необходимости.
2. Для новой логики в `src/lib/` или `src/services/` — **юнит-тесты** (happy path + типичный крайний случай), если не подпадает под исключения из `CLAUDE.md`.
3. Ручная проверка сценария, описанного в задаче (или в `PLAN` для шага).
4. При изменении поведения для пользователя — **CHANGELOG** и при необходимости **STATUS.md**.
5. При новом архитектурном выборе — **ADR** или дополнение существующего (P1-1).

*Lite mode* (мелкие правки) — сокращённый DoD: пункты 1 и 3, без 4–5 при отсутствии видимых изменений.

---

## 5. Порядок внедрения (рекомендуемый)

1. ~~**Неделя 1:** P0 (DoD, `backlog-matrix.md`, коммиты в `CLAUDE.md`)~~ — **сделано.**
2. **Неделя 2–3:** P1-2 CHANGELOG + перенос текущей версии; P1-3 абзац про логику/UI; P1-1 шаблон ADR + 1–2 записи (например: FABRIK/IK цепи, экспорт и камера).
3. **По мере роста команды или релизов:** P2-1, P2-2.

---

## 6. Метрики успеха (качественные)

- Новый участник находит «почему так сделано» за **≤15 минут** (ADR + CLAUDE).
- Конфликтов плана «Step N vs фаза DD» нет: матрица бэклога **актуальна** перед спринтом.
- Доля новых алгоритмов в компонентах без переноса в `lib/` — **снижается** (субъективно на ревью).

---

## 7. Вне объёма этого плана

- Замена существующих инструментов (CI, линтеры), если не согласована отдельно.
- Полный переход на методологию Scrum/Kanban — только если команда решит; здесь — лёгкие артефакты под уже принятый стиль работы.
