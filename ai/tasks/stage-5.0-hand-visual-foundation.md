# Stage 5.0 — Hand Visual Foundation

## 1. Название и цель

**Название:** Stage 5.0 — визуальный фундамент кистей  
**Короткая цель:** добавить у запястий простые визуальные примитивы ладоней, не меняя BODY_25 export и не вводя полноценную механику кистей.  
**Почему это важно для пользователя:** скелет получает читаемый конец руки, а проект получает аккуратную точку расширения для будущих кистей и пальцев.

## 3. Границы задачи

**Входит в задачу:**

- Визуальный примитив правой и левой кисти у `RIGHT_WRIST` / `LEFT_WRIST`.
- Ориентация примитива по направлению `ELBOW -> WRIST`.
- Небольшая асимметрия формы, чтобы читалась ладонь, а не еще одна кость.
- Документация ограничения: примитив не является BODY_25 keypoint и не экспортируется.

**Не входит в задачу:**

- Пальцы.
- Отдельные OpenPose hand keypoints.
- JSON/PNG export новых точек.
- Drag/rotate контроллер кисти.
- Изменение `RigService`, `SkeletonRig`, IK руки или selection flow.
- Повторный клик по запястью для выбора `hand_r` / `hand_l`.

**Файлы/модули, которые можно менять:**

- `poseflow/src/components/skeleton/HandPrimitive.tsx` — новый визуальный компонент.
- `poseflow/src/components/skeleton/Skeleton3D.tsx` — подключение примитива.
- `PLAN.md`, `STATUS.md`, `CHANGELOG.md` — после реализации.

**Файлы/модули, которые не трогаем без отдельной причины:**

- `poseflow/src/services/RigService.ts`
- `poseflow/src/lib/rig/SkeletonRig.ts`
- `poseflow/src/lib/rig/elements.ts`

## 4. Поведение и UX

**Пользовательский сценарий:**

1. Пользователь открывает позу и видит на концах рук небольшие ладони-примитивы.
2. При IK-перетаскивании запястья ладонь едет вместе с ним.
3. При elbow twist, shoulder FK, mirror и пресетах ладонь сохраняет ориентацию по предплечью.
4. Export JSON/PNG остается BODY_25-совместимым и не получает новых keypoints.

**Ожидаемое поведение:**

- Правая и левая ладонь располагаются за запястьем по направлению предплечья.
- Размер пропорционален длине предплечья, но имеет разумные min/max, чтобы не ломаться на экстремальных позах.
- Если локоть или запястье отсутствуют, примитив не рендерится.
- Примитив не перехватывает selection/drag и не мешает `ArmController`.

**Ручная проверка:**

- [ ] T-pose / A-pose: обе кисти видны и выглядят как окончания рук.
- [ ] Wrist IK: ладонь следует за запястьем.
- [ ] Elbow twist: ладонь остается согласованной с предплечьем.
- [ ] Shoulder FK: ладонь следует за рукой.
- [ ] Mirror: левая/правая ладони выглядят симметрично.
- [ ] Export JSON: новых точек нет.
- [ ] Пройти smoke-чеклист из `ai/docs/r3f-smoke-manual-checklist.md`.

## 5. Технический план

**Чистая логика:**

- На первом срезе отдельная pure-логика не нужна, если компонент только вычисляет простую ориентацию меша через `THREE.Vector3`/`Quaternion`.
- Если вычисления формы начнут расти, вынести helper в `poseflow/src/lib/rig/handPrimitive.ts`.

**UI/R3F интеграция:**

- Создать `HandPrimitive.tsx`.
- Использовать геометрию без новых assets: предпочтительно клиновидный `bufferGeometry` или очень короткая приплюснутая box/prism shape.
- Pivot у `WRIST`.
- Направление длины: от `ELBOW` к `WRIST`, с небольшим выносом за запястье.
- Толщина/ширина: меньше предплечья, визуально читается как ладонь.
- Материал: спокойный цвет, согласованный со skeleton, не cyan-гизмо, потому что это не контроллер.

**Сервисы:**

- `RigService`: не менять.
- `PoseService`: не менять.
- `ExportService`: не менять.
