## 2. Актуальная архитектурная линия

Новые фичи должны продолжать текущую модель:

```text
React/R3F controllers
  -> RigService mutations
  -> SkeletonRig state
  -> resolveSkeleton()
  -> PoseData for rendering/export
```

Координатный контракт (ADR 0004):

```text
skeleton/local pose + rootPosition/rootRotation -> world PoseData
world/screen input -> local frame delta -> SkeletonRig mutation
```

`SkeletonRig` хранит позу в системе координат манекена. `rootPosition` и `rootRotation` — единственный transform манекена относительно рабочего пространства. Узел `MID_HIP` / 8 управляет этим root transform.

Правила:

- Не возвращать старый прямой `updateJoint`/FK-drag подход в `PoseService`.
- Root-гизмо узла `MID_HIP` / 8 должно следовать за `rootRotation`: стрелки трансляции работают вдоль локальных осей скелета, кольца вращают вокруг локальных осей скелета через круговой drag.
- Не восстанавливать удалённые экспериментальные модули без отдельного решения.
- Нетривиальную математику держать в `src/lib/rig/`, `src/lib/solvers/` или `src/services/`.
- Компоненты в `src/components/controllers/` должны оркестрировать R3F, pointer events и вызовы сервиса, а не содержать основную механику solver-логики.
- Любой controller/input код должен переводить screen/world drag в соответствующий local frame до применения локального `RigService` delta.
- Ручные таблицы инверсий допустимы только как диагностика; целевое решение — frame-based пересчёт.
- Все стрелочные элементы гизмо должны использовать единый cyan-цвет `#00ccff` (как у гизмо локтя/руки).
- Гизмо spine/neck должно ориентироваться по torso, а не по мировым осям: вверх по позвоночнику, горизонталь по линии плеч.
- Чувствительность mouse drag для гизмо должна быть приведена к единой настройке в параметрах приложения; это future task, не часть текущей починки `HeadController`.
- Hit-зоны гизмо должны быть приведены к единой настройке в параметрах приложения; это отдельная future task и не часть текущей починки `HeadController`.
- Перед правкой controller/gizmo сначала фиксировать механику: pivot, ось вращения, плоскость движения, расположение гизмо, затронутые суставы/кости, стороны манекена и ожидаемый знак mouse drag для основных видов камеры.
- Когда пользователь говорит "правая/левая" рука или нога, это означает сторону с точки зрения скелета/манекена. При правках кода отдельно сверять это с BODY_25 side key (`r` / `l`), потому что визуальная сторона в текущем виде камеры может казаться зеркальной.
- Если задача связана с инверсией мыши или видом камеры, сначала делать короткую таблицу поведения, а не менять знаки по одному наугад.
- Новая чистая логика требует unit-тестов: happy path + один вероятный edge case.

Ключевые файлы для ориентира:

| Файл | Роль |
|------|-----|
| `poseflow/src/services/RigService.ts` | Основные операции над rig, undo/redo, уведомления |
| `poseflow/src/services/PoseService.ts` | Совместимость для экспорта, пресетов и старых вызовов |
| `poseflow/src/components/Canvas3D.tsx` | Интеграция сцены, выделения и контроллеров |
| `poseflow/src/components/controllers/ArmController.tsx` | Текущий образец нового контроллера |
| `poseflow/src/lib/rig/armIK.ts` | Текущий образец вынесенной чистой логики |
| `poseflow/src/lib/feature-flags/registry.ts` | Актуальный список feature flags |
| `STATUS.md` | Актуальный пользовательский статус продукта |

## 3. Известные ограничения

- Нет полноценных контроллеров кистей: есть только визуальный `HandPrimitive` без пальцев и drag-контроллера.
- Нет контроллеров ног.
- Нет продвинутого управления стопами: пальцы/пятка пока вращаются только как жёсткая группа вокруг лодыжки.
- Нет управления несколькими скелетами.
- Нет центра тяжести.

## 4. Roadmap

### Stage 4.2 — ShoulderController

**Статус:** выполнено 2026-04-28.  
**Цель:** дать отдельное управление плечом без разрушения текущей IK руки и spine-цепочки.

**Подготовка:** параметры первого среза и task brief зафиксированы в `ai/tasks/stage-4.2-shoulder-controller.md`.

Реализовано:

- `shoulder_r` / `shoulder_l` в selection model.
- `ShoulderController` с 4 стрелками FK у выбранного плеча.
- `shoulderFK.ts` с pure-логикой лимитов и применения delta к rotation-tree.
- `RigService`: `applyShoulderRaise`, `applyShoulderForward`.
- Unit-тесты pure-логики, selection и сервисного поведения.

Минимальный срез:

- Выделение: клик по `RIGHT_SHOULDER` / `LEFT_SHOULDER` выбирает `shoulder_r` / `shoulder_l`.
- UI: компактный стрелочный контроллер у выбранного плеча: вверх/вниз и вперёд/назад.
- Логика: операции идут через `RigService`; чистая математика выносится в `src/lib/rig/`.
- Поведение: изменение плеча сохраняет длины плечо→локоть и локоть→запястье, не ломает `ArmController`.
- Undo/redo: один drag даёт один undo-снимок через `beginDrag()`.
- Тесты: новая pure-логика + сервисные методы, минимум happy path и крайний случай.
- Ручная проверка: обе стороны, undo/redo, совместимость с elbow twist и wrist IK.

Параметры первого среза:

- Оси: raise/lower и forward/back для движения точки `RIGHT_SHOULDER` / `LEFT_SHOULDER` вокруг `NECK`.
- Лимиты: raise/lower `-75°..+120°`, forward/back `-60°..+90°`.
- Поведение руки: рука следует за плечом; IK-lock кисти не входит в Stage 4.2.
- Важное ограничение rig: вращение плеча должно менять сегмент `NECK -> SHOULDER` (`1-2` или `1-5`), а не только верхнюю руку `SHOULDER -> ELBOW`.

### Stage 5.0 — Hand Visual Foundation

**Статус:** выполнено 2026-04-29.  
**Цель:** добавить визуальный примитив кисти у каждого запястья как фундамент для будущей полноценной работы с кистями.

Подготовка: параметры первого среза зафиксированы в `ai/tasks/stage-5.0-hand-visual-foundation.md`.

Реализовано:

- `HandPrimitive` у `RIGHT_WRIST` / `LEFT_WRIST`.
- Ориентация по направлению предплечья `ELBOW -> WRIST`.
- Примитивы не участвуют в selection/drag и не меняют BODY_25 export.
- `RigService`, `SkeletonRig`, selection flow и IK-логика руки не изменялись.

Минимальный срез:

- Визуальный `HandPrimitive` у `RIGHT_WRIST` / `LEFT_WRIST`.
- Примитив ориентируется по направлению предплечья `ELBOW -> WRIST` и следует за arm IK, elbow twist, shoulder FK, mirror и пресетами.
- Примитив не является BODY_25 keypoint, не участвует в OpenPose JSON/PNG export и не добавляет helper-точки в `PoseData`.
- Selection flow не меняется: клик по запястью по-прежнему выбирает `arm_r` / `arm_l`.
- `RigService`, `SkeletonRig` и IK-логика руки не меняются в этом срезе.

Ограничения:

- Без пальцев.
- Без отдельного drag/rotate контроллера кисти.
- Без повторного клика по wrist для выбора `hand_r` / `hand_l`.
- Полноценный `HandController` переносится на будущий этап после отдельного дизайна формата helper-точек/ориентации кисти.

### Future — Arm natural limits

**Статус:** отложено.  
**Цель:** добавить естественные ограничения для управления руками, чтобы IK запястья, shoulder FK и elbow twist не выводили руку в анатомически невозможные положения.

Почему не делаем в текущем проходе:

- это отдельная биомеханическая модель, а не простая правка гизмо;
- нужно согласовать лимиты плечевого пояса, локтя и, позднее, кисти;
- ограничения должны одинаково работать при shoulder FK, wrist IK, elbow twist, mirror и после поворота всего манекена;
- сначала лучше закрыть текущий проход по привязке гизмо к координатам скелета.

Принятый предварительный подход:

- Делать отдельный модуль чистой логики `armLimits.ts` по аналогии с `legIK.ts`, но с отдельной моделью руки.
- Не ограничиваться только углом локтя: ограничения должны учитывать плечевой пояс, плечевой сустав, локоть и осевой twist костей.
- При достижении лимита движение должно останавливаться на последней допустимой позе, без скачка к "исправленной" позе.
- Все проверки строить в координатах скелета/манекена, а не в мировых координатах.

Что нужно учитывать:

- `ShoulderController` управляет плечевым поясом: двигает BODY_25 плечо `2` / `5` вокруг `NECK` (`1`).
- `ArmController` управляет рукой: wrist IK через `applyArmIK` и локтевой swivel через `applyElbowTwist`.
- Ограничения плечевого пояса и верхней руки не одно и то же: точка плеча может двигаться, а кость `SHOULDER -> ELBOW` имеет свои лимиты направления.
- Нужно ограничить сгибание локтя: базовый первый вариант — запрет переразгиба назад и flexion примерно до `145..150°`.
- Нужно ограничить направление верхней руки `2-3` / `5-6` в локальной системе торса/плеча.
- Нужно ограничить вращение верхней руки вокруг своей оси в плечевом суставе: axial twist костей `2-3` и `5-6`.
- Нужно ограничить вращение предплечья вокруг своей оси в локте: axial twist костей `3-4` и `6-7`.
- Для axial twist использовать swing-twist decomposition, чтобы отделять "куда смотрит кость" от "как она провернута вокруг своей оси".

Порядок реализации:

1. Создать task brief и `armLimits.ts` с тестами на чистую математику.
2. Реализовать локоть flexion/extension и подключить к `applyArmIK` в режиме stop-at-limit.
3. Добавить ограничение направления верхней руки `SHOULDER -> ELBOW`.
4. Добавить axial twist limit для верхней руки `2-3` / `5-6`.
5. Добавить axial twist limit для предплечья `3-4` / `6-7`.
6. Проверить `applyElbowTwist`: он должен учитывать и swivel-сектор локтя, и axial twist лимиты.
7. Проверить совместимость с `ShoulderController`: shoulder FK не должен оставлять руку в невозможном состоянии.
8. После каждого шага запускать unit-тесты и ручную проверку обеих рук, undo/redo и поворота всего манекена.

### Future — Leg limits refinement

**Статус:** отложено.  
**Цель:** пересмотреть ограничения ног как отдельную solver/UX-задачу, чтобы ankle IK, knee twist и hip limits не конфликтовали при сложных позах.

Текущие ограничения уже есть, но требуют второго прохода:

- knee bend: вперёд до 85°, назад 0°;
- hip direction limits: вперёд/назад и наружу/внутрь;
- ankle IK останавливается, если цель требует выхода бедра за лимиты;
- knee twist сохраняется при повторном ankle IK.

Что нужно проработать отдельно:

- где именно пользователь ощущает "упирание" или неожиданный отказ движения;
- как совмещать knee twist с hip limits без скачков и без потери выбранной twist-плоскости;
- нужна ли визуальная индикация достижения лимита;
- стоит ли разделить мягкие и жёсткие лимиты;
- как вести себя при уже невозможной позе после ручного twist: сохранять последнюю допустимую позу, частично проецировать цель или явно блокировать дальнейшее движение.

### Stage 7 — FootController

**Статус:** выполнено 2026-04-30 первым срезом.  
**Цель:** управление стопой как жёсткой группой `{ankle, big toe, small toe, heel}`.

Реализовано:

- Pivot у `ANKLE`.
- Pitch/yaw/roll стопы с лимитами: pitch `-50°..+30°`, yaw `-35°..+35°`, roll `-25°..+25°`.
- `FootController`: pitch "на себя/от себя" — cyan-стрелки вверх/вниз из центра отрезка между пальцами; yaw "вправо/влево" — стрелки наружу из большого и малого пальца с cyan-отрезком между ними; roll — cyan-окружность в середине пальцев, перпендикулярная линии `HEEL -> midpoint(BIG_TOE, SMALL_TOE)`.
- Foot rotation строит плоскости относительно голени: yaw в плоскости, перпендикулярной голени; pitch в плоскости, проходящей через голень и середину пальцев.
- Mouse drag для foot-гизмо согласован ручной таблицей: pitch одинаков для обеих стоп, yaw и roll переключают знак по локальной стороне стопы (носок/пятка), roll работает угловым drag как "колёсико".
- Hit-зона roll-кольца уменьшена, чтобы не перехватывать yaw-стрелки.
- Отдельный слой `footAngles` / `footRotations` в `SkeletonRig`, чтобы вращение стопы не двигало `ANKLE`.
- OpenPose export остаётся совместимым: меняются только BODY_25 позиции toe/heel.
- Unit-тесты pure-логики и `RigService`.

- Возможное решение для конфликта: контекстное меню по короткому right-click без drag, а right-drag оставить для камеры.

### Future — Scene save/load

**Статус:** дальний план, после отладки основных моделей данных.  
**Цель:** добавить сохранение и загрузку полной сцены PoseFlow.

Почему откладываем:

- Формат файла нельзя фиксировать до стабилизации основных частей модели.
- Ограничения рук/ног, полноценные кисти, helper-точки, ориентации кистей, несколько скелетов и возможные scene-level настройки могут существенно повлиять на состав данных.
- Ранний формат приведёт к миграциям и совместимости раньше, чем мы уверены в структуре сцены.

Что будущий формат должен учитывать:

- Версию формата и миграции.
- Один или несколько `SkeletonRig`.
- Root transform каждого манекена и позу в координатах скелета.
- Настройки сцены, камеры, export frame и, возможно, viewport/UI-предпочтения.
- Расширенные данные вне BODY_25: кисти, helper-точки, будущие контроллеры и метаданные.
- Импорт/экспорт без привязки к текущему `localStorage`.

Предварительное решение:

- Проектировать формат после стабилизации arm limits, leg limits refinement и решения по полноценным кистям.
- Не смешивать этот формат с OpenPose BODY_25 export: это разные задачи.

### Future R&D — Image-to-3D skeleton via ComfyUI

**Статус:** сильно позже, отдельное исследование.  
**Цель:** построить pipeline, который превращает исходное изображение человека в редактируемый 3D-скелет PoseFlow.

Предварительная цепочка:

```text
source image -> flat 2D skeleton image/keypoints -> inferred 3D skeleton pose -> PoseFlow SkeletonRig
```

Почему это R&D:

- Нужно выбрать модель/узлы ComfyUI для извлечения 2D позы.
- Нужно решить, работаем ли через изображение скелета, keypoints JSON или оба формата.
- Нужно исследовать 2D-to-3D pose lifting: восстановление глубины неоднозначно и требует эвристик или отдельной модели.
- Нужно сопоставить внешний формат позы с BODY_25 и внутренним `SkeletonRig`.
- Нужно определить, как пользователь будет исправлять неоднозначности: вид сбоку, глубина конечностей, пересечения, рост/пропорции.
- Требуется отдельный task brief и, вероятно, ADR до реализации.

Возможные этапы исследования:

1. ComfyUI/OpenPose extraction: получить BODY_25-compatible 2D keypoints из изображения.
2. Import 2D pose into PoseFlow as flat pose.
3. 2D-to-3D lifting prototype: восстановить глубину и root orientation.
4. UI для ручной коррекции глубины и стороны конечностей.
5. Сохранение результата в будущий scene save/load формат.

### Future R&D — Character mesh import and retargeting

**Статус:** сильно позже, отдельное исследование.  
**Цель:** импортировать подготовленного в другой программе персонажа и привязать его к скелету PoseFlow для более выразительной визуализации.

Открытые вопросы:

- Какие форматы поддерживать первым срезом: `glTF/GLB`, `FBX`, `OBJ`, другие.
- Нужен ли уже rigged/skinned персонаж или допускается статический mesh.
- Если персонаж rigged: как сопоставлять его кости с BODY_25 / `SkeletonRig`.
- Если персонаж animated: импортируем ли анимацию или только bind pose + skin weights.
- Как хранить scale, bind pose, bone mapping, mesh assets и материалы в scene save/load формате.
- Как решать несовпадение пропорций между персонажем и текущим манекеном.
- Как сохранить offline desktop workflow и не раздувать приложение тяжёлыми зависимостями без необходимости.

Предварительные рекомендации:

- Для первого исследования приоритетнее `glTF/GLB`, потому что это современный web/Three.js-friendly формат со skinned mesh.
- `OBJ` можно рассматривать только как статический mesh без скелета; для rigged/animated персонажей он сам по себе недостаточен.
- Не начинать реализацию до проектирования scene save/load и решения по полноценному внутреннему формату скелета/кистей.
- Требуется отдельный R&D brief, тестовые ассеты и ADR по формату импорта/retargeting.

## 6. Рабочий протокол для вайбкодинга

Перед каждой задачей:

1. Выполнить current-state probe: `git status --short`, `STATUS.md`, этот `PLAN.md`, `AGENTS.md`, релевантные файлы кода.
2. Если документация противоречит коду, считать код и `STATUS.md` более актуальными; противоречие отметить в брифе или исправить отдельной docs-правкой.
3. Для non-Lite feature task заполнить краткий бриф по `ai/docs/feature-task-template.md`, включая результат current-state probe.
4. Сформулировать маленький вертикальный срез.
5. Уточнить параметры, если без них можно выбрать неверную механику тела.
6. Для 3D-гизмо до правки описать расчёт: pivot, оси, плоскости, расположение контролов, правая/левая сторона с точки зрения манекена, таблица знаков mouse drag по видам спереди/сзади/справа/слева.

Во время реализации:

1. Не делать drive-by refactor.
2. Сначала вынести чистую механику, затем подключить UI-контроллер.
3. Сохранять undo/redo паттерн `beginDrag()` один раз на pointer down.
4. Проверять, что `Canvas3D` остаётся тонким интеграционным слоем.
5. Если ручная проверка показывает "всё наоборот" или "бардак со знаками", остановиться и пересчитать таблицу видов, а не продолжать серию точечных инверсий.

Перед завершением:

1. Запустить релевантные unit-тесты или весь `npm test`.
2. Для UI/R3F/drag/export viewport изменений выполнить smoke checklist из `ai/docs/r3f-smoke-manual-checklist.md`.
