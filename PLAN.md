**Версия продукта:** v0.3.0-dev  
**Дата актуализации:** 2026-04-28

Этот файл — короткий источник правды для следующих итераций. Если он противоречит коду, приоритет у кода и `STATUS.md`; после проверки расхождение нужно исправить здесь.

## 1. Текущее состояние

### Готово

- Базовый редактор BODY_25: скелет, кости, пресеты, камера, экспорт PNG/JSON, рамка экспорта, настройки, Electron/FastAPI.
- Rotation-tree архитектура:
  - `poseflow/src/lib/rig/` — `SkeletonRig`, `RestPose`, `VirtualChain`, `resolveSkeleton`, `inverseFK`, `elements`.
  - `RigService` — первичный источник истины позы.
  - `PoseService` — совместимая тонкая обёртка над `RigService`.
  - `SelectionService` — выделение элемента тела кликом.
- Контроллеры DesignDoll-style:
  - Stage 1: `PelvisController`, `SpineController`.
  - Stage 2: `NeckController`.
  - Stage 3: `HeadController`.
  - Stage 4.1: `ArmController` — IK запястья через FABRIK + скручивание локтя.
- Mini-view интегрирован через `USE_MINI_VIEW`.
- DI container интегрирован в `App.tsx` / `main.tsx`.
- Feature flags сохранены только для актуальных возможностей. Старые флаги предыдущей попытки DesignDoll-рефакторинга удалены.

### Проверенный контур качества

- `npm test` из `poseflow/`: 174 unit-теста проходят.
- `npx vite build` из `poseflow/`: Vite-сборка проходит.
- `npm run typecheck` из `poseflow/`: проверяет React/app и Electron/Vite configs отдельными `tsconfig`.
- `npm run build` из `poseflow/`: typecheck + Vite/Electron build.
- `npm run verify` из `poseflow/`: typecheck + unit-тесты + Vite/Electron build.

## 2. Актуальная архитектурная линия

Новые фичи должны продолжать текущую модель:

```text
React/R3F controllers
  -> RigService mutations
  -> SkeletonRig state
  -> resolveSkeleton()
  -> PoseData for rendering/export
```

Правила:

- Не возвращать старый прямой `updateJoint`/FK-drag подход в `PoseService`.
- Не восстанавливать удалённые экспериментальные модули без отдельного решения.
- Нетривиальную математику держать в `src/lib/rig/`, `src/lib/solvers/` или `src/services/`.
- Компоненты в `src/components/controllers/` должны оркестрировать R3F, pointer events и вызовы сервиса, а не содержать основную механику solver-логики.
- Новая чистая логика требует unit-тестов: happy path + один вероятный edge case.

Ключевые файлы для ориентира:

| Файл | Роль |
|------|-----|
| `poseflow/src/services/RigService.ts` | Основные операции над rig, undo/redo, уведомления |
| `poseflow/src/services/PoseService.ts` | Совместимость для экспорта, пресетов и старых вызовов |
| `poseflow/src/components/Canvas3D.tsx` | Интеграция сцены, выделения и контроллеров |
| `poseflow/src/components/controllers/ArmController.tsx` | Текущий образец нового контроллера |
| `poseflow/src/lib/rig/armIK.ts` | Текущий образец вынесенной чистой логики |
| `poseflow/src/lib/feature-flags/registry.ts` | Актуальный список feature flags |
| `STATUS.md` | Актуальный пользовательский статус продукта |

## 3. Известные ограничения

- Нет отдельного `ShoulderController`: плечи пока зависят от spine и IK рук.
- Нет контроллеров кистей.
- Нет контроллеров ног.
- Нет контроллеров стоп.
- Нет управления несколькими скелетами.
- Нет центра тяжести.

## 4. Roadmap

### Stage 4.2 — ShoulderController

**Статус:** следующий продуктовый шаг.  
**Цель:** дать отдельное управление плечом без разрушения текущей IK руки и spine-цепочки.

Минимальный срез:

- Выделение: клик по `RIGHT_SHOULDER` / `LEFT_SHOULDER` выбирает `shoulder_r` / `shoulder_l`.
- UI: компактный контроллер у выбранного плеча, визуально согласованный с текущими кольцами.
- Логика: операции идут через `RigService`; чистая математика выносится в `src/lib/rig/`.
- Поведение: изменение плеча сохраняет длины плечо→локоть и локоть→запястье, не ломает `ArmController`.
- Undo/redo: один drag даёт один undo-снимок через `beginDrag()`.
- Тесты: новая pure-логика + сервисные методы, минимум happy path и крайний случай.
- Ручная проверка: обе стороны, undo/redo, совместимость с elbow twist и wrist IK.

Перед реализацией нужно коротко зафиксировать параметры:

- какие оси плеча нужны в первом срезе;
- лимиты углов;
- должна ли кисть сохранять мировую позицию при вращении плеча или следовать за рукой.

### Stage 5 — HandController

**Статус:** ожидает Stage 4.2.  
**Цель:** управление ориентацией кистей в пределах BODY_25, без добавления новых OpenPose keypoints.

Минимальный срез:

- Контроллер у `RIGHT_WRIST` / `LEFT_WRIST`.
- Поворот кисти влияет на toe/hand-отсутствующие точки не должен симулироваться фейковыми суставами в BODY_25.
- Если нужны дополнительные визуальные helper-точки, они не должны попадать в OpenPose export.

### Stage 6 — LegController

**Статус:** ожидает.  
**Цель:** FABRIK IK для ноги: бедро→колено→лодыжка.

Минимальный срез:

- Сфера на лодыжке для camera-plane drag.
- Бедро фиксировано, колено решается FABRIK.
- Скручивание колена по образцу elbow twist, если оно не усложняет первый срез.
- Чистая логика рядом с `armIK.ts` или в обобщённом модуле цепей, если это реально уменьшит дублирование.

### Stage 7 — FootController

**Статус:** ожидает Stage 6.  
**Цель:** управление стопой как жёсткой группой `{ankle, big toe, small toe, heel}`.

Минимальный срез:

- Pivot у ankle или heel — выбрать перед реализацией.
- Pitch/roll/yaw стопы с биомеханически разумными лимитами.
- OpenPose export остаётся совместимым: меняются только BODY_25 позиции.

### Multi-skeleton

**Статус:** отложено.  
**Важно:** текущий `PoseService` больше не содержит полноценную архитектуру `skeletons[]`; старые записи плана об этом устарели.

Перед реализацией нужен отдельный дизайн:

- где хранится список `SkeletonRig`;
- как устроены undo/redo для нескольких фигур;
- как selection связывается с active skeleton;
- как экспорт выбирает одну или несколько фигур.

### Center of Gravity

**Статус:** отложено.  
**Важно:** актуального `CenterOfGravity.tsx` в коде нет; старые записи плана об уже созданном компоненте устарели.

Перед реализацией нужен отдельный дизайн:

- какие суставы входят в верхнюю/нижнюю группу;
- является ли CoG инструментом позирования или только визуальной подсказкой;
- как CoG взаимодействует с `RigService` и undo/redo.

## 6. Рабочий протокол для вайбкодинга

Перед каждой задачей:

1. Выполнить current-state probe: `git status --short`, `STATUS.md`, этот `PLAN.md`, `AGENTS.md`, релевантные файлы кода.
2. Если документация противоречит коду, считать код и `STATUS.md` более актуальными; противоречие отметить в брифе или исправить отдельной docs-правкой.
3. Для non-Lite feature task заполнить краткий бриф по `ai/docs/feature-task-template.md`, включая результат current-state probe.
4. Сформулировать маленький вертикальный срез.
5. Уточнить параметры, если без них можно выбрать неверную механику тела.

Во время реализации:

1. Не делать drive-by refactor.
2. Сначала вынести чистую механику, затем подключить UI-контроллер.
3. Сохранять undo/redo паттерн `beginDrag()` один раз на pointer down.
4. Проверять, что `Canvas3D` остаётся тонким интеграционным слоем.

Перед завершением:

1. Запустить релевантные unit-тесты или весь `npm test`.
2. Для UI/R3F/drag/export viewport изменений выполнить smoke checklist из `ai/docs/r3f-smoke-manual-checklist.md`.
3. Сообщить, какие проверки прошли и какие не запускались.
